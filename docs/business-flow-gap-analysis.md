## 各フェーズのギャップ詳細

### ① 入荷・検品 — ⚠️ 部分対応

**前回❌ → 今回⚠️**: ドライバー出勤管理UIを追加。

| 実務でやること | アプリの状態 |
|--------------|------------|
| マニフェスト (CSV/EDI) の受領・取り込み | ❌ 対応なし。DB直接投入が前提 |
| 物理荷物とマニフェストの突合 | ❌ 対応なし |
| 破損・欠品の記録 | ❌ 対応なし |
| ドライバー出勤確認 | ✅ **DriverAttendancePanel** — ドライバーごとに出勤/欠勤トグル + 一括保存 (`POST /api/plan/driver-attendance`) |

**残りの改善案**:
- 荷物インポートAPI (`POST /api/plan/packages/import`) — CSV取り込み
- *デモとしての優先度*: **低**。デモではデータ事前投入で十分

### ② 計画・ルート生成 — ✅ 完成度高い

| 実務でやること | アプリの状態 |
|--------------|------------|
| 荷物をドライバーに割り当て | ✅ 自動割当 + 手動割当ボード |
| ルート順序の最適化 | ✅ 貪欲法 + H3コスト行列 + MLリスク考慮 |
| 時間指定の優先配置 | ✅ 午前/午後/夜間/フレキシブルの4プール分割 |
| 手動での順序調整 | ✅ ドラッグ&ドロップ |
| ルート確定 | ✅ `delivery_status` を `assigned` に一括更新、`trip_number`・`stop_order` も保存 |
| N便ルート生成 | ✅ **ドライバーの`max_trips`に応じた動的N便** (固定2便→可変N便に改善) |
| 置き配/対面配達区分 | ✅ `delivery_method` (drop_off/face_to_face) による滞在時間差異考慮 |
| 確定後の緊急追加 | ❌ 確定後に荷物を追加する手段がない |

### ④ 出庫 — ❌ 未対応

ステータスが `loaded` から直接 `in_transit` に遷移。出庫という明示的な操作はない。

| 実務でやること | アプリの状態 |
|--------------|------------|
| 出発確認・時刻記録 | ❌ |
| ルートをドライバー端末に送信 | ❌ ドライバーアプリなし |
| 出庫記録 | ❌ |

*デモとしての優先度*: **低**。GPS追跡が始まれば出庫は暗黙的にわかる。

## delivery_status のステータス遷移 — 現状

### 実装済みのステータス

```
pending → assigned → loaded → in_transit → delivered
                                          → absent → (次便で redelivery)
                                          → failed
                   (日次締め時)
                   → returned (DailyClosePanel で一括処理)
```

**改善前との差分**:
- `loaded` ステータス追加 (LoadConfirmButton で `assigned → loaded`)
- `returned` ステータス追加 (DailyClosePanel で未配達を一括返却)
- `trip_number` / `stop_order` カラム追加 (N便管理、再調整永続化)

### DB スキーマ変更 (ALTER TABLE)

```sql
ALTER TABLE delivery_status
  ADD COLUMN trip_number INTEGER DEFAULT 1,
  ADD COLUMN stop_order INTEGER;

ALTER TABLE delivery_status DROP CONSTRAINT IF EXISTS delivery_status_status_check;
ALTER TABLE delivery_status ADD CONSTRAINT delivery_status_status_check
  CHECK (status IN ('pending','assigned','loaded','in_transit','delivered','absent','failed','returned'));

CREATE INDEX idx_delivery_status_trip ON delivery_status (driver_id, date, trip_number);
```

> 詳細設計: `docs/flexible-trip-design.md`

## データアクセスパスの改善 (2026-03-12)

Plan画面のリスクマップ・マンション密集レイヤーに **Snowflake直接 / pg_lake Foreign Table (ft_*) の切り替え** を実装。サーバーサイドのクエリ実行時間をリアルタイム表示。

| API | SF 直接 | ft_* (pgQuery) | 切替パラメータ |
|-----|---------|---------------|-------------|
| `/api/plan/risk-map` | ✅ `ANALYTICS.RISK_SCORES` | ✅ `ft_risk_scores` | `?source=sf\|pg` |
| `/api/plan/building-density` | ✅ `ANALYTICS.BUILDING_ATTRIBUTES` + `DELIVERY_HISTORY` | ✅ `ft_building_attributes` + `ft_delivery_history` | `?source=sf\|pg` |

## デモとして「見せ方」を変えるだけで解決するもの

| ギャップ | デモでの見せ方 |
|---------|-------------|
| 入荷・検品がない | 荷物データは基幹システム（配送管理で採番）から自動連携される前提 |
| 出庫ボタンがない | ドライバーのGPS移動開始で自動的に出庫と判定。作っても良いが非クリティカル |
| ドライバー端末がない | ハード側のシステムからデータを貰う前提。Phase 2 でモバイルアプリ追加も可 |
| 日報PDF出力がない | CSV出力 + BI ツール連携で対応可能。あっても良いが非クリティカル |

## 追加ギャップ候補（アイデア記録・実装未定）

> 以下は業務観点で洗い出したもの。現時点では優先度は高くない。

### 配送所管理者（所長）視点

| ギャップ | 深刻度 | 説明 |
|---------|--------|------|
| 時間指定違反の可視化がない | 中 | SLA違反は再調整時に表示されるが「今日何件違反したか」の集計ビューがない |
| 不在時の対応記録がない | 低 | 不在票投函→再配達受付→翌日組み込みのフロー。現状は `absent` フラグのみ |
| 荷物個別の配達完了時刻追跡がない | 中 | `completed_at` はDBにあるが、Review画面にドリルダウンがない |
| ドライバー間の荷物手動移動UIがない | 低 | 緊急離脱時の自動再割当はあるが、日常的な手動移動がない |
| 過積載アラートがない | 低 | 重量/体積は表示されるがチェックしない |

### 配送所スタッフ（事務員・仕分け担当）視点

| ギャップ | 深刻度 | 説明 |
|---------|--------|------|
| 電話対応の記録がない | 中 | 荷受人から「今どこ？」「時間変更したい」の電話。メモする場所がない |
| 再配達受付フローがない | 中 | 不在後の再配達をWeb/電話で受付→翌日組み込みの導線がない |
| 仕分けエリアの表示がない | 低 | ドライバー別の仕分けレーン/棚番号の物理配置情報がない |

### エリア統括（複数配送所マネージャー）視点

| ギャップ | 深刻度 | 説明 |
|---------|--------|------|
| 配送所横断のダッシュボードがない | 高 | 単一depot前提。5〜10拠点の比較ができない |
| 配送所間の荷物融通がない | 中 | 隣接配送所への荷物移管フローがない |
| ドライバーの稼働率比較がない | 中 | 配送所ごとの1人あたり配達数・走行距離の比較ができない |
| 翌日の人員配置支援がない | 中 | 需要予測はあるが「人が足りない」アラートがない |
